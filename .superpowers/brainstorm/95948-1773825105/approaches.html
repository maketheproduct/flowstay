Architecture Approach

How should meeting recording and the Library integrate into Flowstay's existing module structure?

A

New SPM Module: FlowstayMeetings

Recommended. Clean separation of concerns with a dedicated module.

Module Architecture
Package.swift
├── Flowstay          (app entry point)
├── FlowstayCore      (existing: dictation, personas, AI providers)
├── FlowstayUI        (existing: menu bar, settings, overlay)
├── FlowstayPermissions (existing: onboarding)
│
├── FlowstayMeetings  ← NEW MODULE
│   ├── Audio/
│   │   ├── SystemAudioCapture    (ScreenCaptureKit)
│   │   ├── MicrophoneCapture     (AVAudioEngine)
│   │   └── AudioMixer            (combines both streams)
│   ├── Detection/
│   │   ├── MeetingAppDetector    (NSWorkspace observation)
│   │   └── MeetingSession        (lifecycle management)
│   ├── Transcription/
│   │   ├── MeetingTranscriber    (FluidAudio pipeline)
│   │   └── SpeakerDiarization    (label who spoke)
│   ├── Notes/
│   │   ├── NoteGenerator         (AI-powered, uses AIProviderProtocol)
│   │   └── MeetingTemplate       (1:1, Stand-Up, Generic, Custom)
│   └── Models/
│       └── SwiftData models
│
└── FlowstayLibrary   ← NEW MODULE
    ├── LibraryWindow         (main window)
    ├── MeetingDetailView     (transcript + notes)
    ├── TranscriptionListView (existing history, migrated)
    └── LibrarySidebar        (navigation)

Pros

  • Clean boundaries — meeting code doesn't bloat FlowstayCore
  • Independent compilation and testing
  • Library window gets its own module for future expansion (chat, screenshots)
  • Reuses AIProviderProtocol from FlowstayCore

Cons

  • More SPM modules to manage
  • Need to define clean interfaces between modules
B

Extend Existing Modules

Add meeting features directly into FlowstayCore and FlowstayUI.

Module Architecture
Package.swift
├── Flowstay          (app entry point)
├── FlowstayCore      (extended)
│   ├── ... existing code ...
│   ├── Meetings/           ← NEW DIRECTORY
│   │   ├── SystemAudioCapture
│   │   ├── MeetingAppDetector
│   │   ├── MeetingTranscriber
│   │   ├── NoteGenerator
│   │   └── MeetingTemplate
│   └── Storage/            ← NEW DIRECTORY
│       └── SwiftData models
│
├── FlowstayUI        (extended)
│   ├── ... existing views ...
│   └── Library/            ← NEW DIRECTORY
│       ├── LibraryWindow
│       ├── MeetingDetailView
│       └── LibrarySidebar
│
└── FlowstayPermissions (existing)

Pros

  • Fewer modules, simpler dependency graph
  • Direct access to existing services (AppState, etc.)
  • Faster to get started

Cons

  • FlowstayCore grows significantly — mixing dictation and meetings
  • Harder to reason about boundaries as features grow
  • All of Core recompiles when meeting code changes
