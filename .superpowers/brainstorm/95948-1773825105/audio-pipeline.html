Audio Pipeline Design

How meeting audio flows from capture to transcribed text

Key Insight: Free Speaker Diarization

Since we capture mic and system audio as separate streams, we get speaker labels for free — no ML diarization needed. The mic stream = you, the system audio stream = remote participants.

Technical Details

SystemAudioCapture

  • SCShareableContent to find meeting app
  • SCContentFilter with app-level filter
  • SCStream for audio-only capture
  • SCStreamConfiguration: audio only, no video
  • Requires screen capture permission (TCC)
  • Resample from app's rate → 16kHz mono

MicrophoneCapture

  • AVAudioEngine with input node tap
  • Separate from FluidAudio's internal mic handling
  • Or: feed FluidAudio directly for mic stream
  • Already have microphone permission from dictation
  • 16kHz mono PCM (FluidAudio's native format)
  • Reuses existing audio infrastructure where possible

Open Question: FluidAudio Instances

FluidAudio/Parakeet is designed for single-stream input. For two simultaneous streams, we have two options:

Option 1: Two FluidAudio instances

Run two Parakeet models in parallel — one per stream. Uses ~2x memory but gives clean speaker separation.

Option 2: Alternating chunks

Feed chunks from each stream to a single instance, alternating. Less memory but may lose context between switches.
